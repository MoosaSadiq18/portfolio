The Only Docker Guide You Need

The actual problem

Before I go into the details of Docker, I want to explain the problem developers faced that led to its creation.

Suppose you are a backend developer working on a Node.js project and you want your teammate who is a frontend developer to run the service. You send him the repo link. What happens next is that he might face issues running that service because he does not have those backend related dependencies in his system.

Even if he installs them, he may face version conflicts.

Example:

  • Your service uses an older version of Node.js
  • Your teammate installs the latest version

This leads to the classic developer problem: 

Works on my machine.
---

Use Virtual Machines?

Before containerization got popular, Virtual Machines were used to solve this issue.

Typical workflow:

  • Create a VM with its own operating system
  • Install all dependencies and configure the project
  • Send the VM image to your teammate

VM file extensions looked like this:

.vmx
.qcow2

Your teammate would run the VM and the service would work perfectly.

---

Wait, no need for containers then?

Now here is the problem with VMs.

A VM carries its own operating system, kernel, libraries, and components even if your application doesn't use them.

A typical VM image can easily be: 

20GB - 50GB

It also consumes a large amount of RAM.

Containers solve the same problem but in a lightweight way.

  • Containers share the host OS kernel
  • No separate OS required
  • Images are usually MBs instead of GBs

So instead of sending a **20GB VM**, you send a **200MB container image**.

---

So… What Even Is Docker?

Think of Docker as a box that contains everything your application needs to run.

Every time you run it, Docker creates a container instance from that box.

---

Dockerfile

A Dockerfile contains instructions for building an image.

It defines things like:

  • Base image
  • Dependencies
  • Commands to run

Example Dockerfile:


FROM openjdk:17

WORKDIR /app

COPY target/app.jar app.jar

CMD ["java","-jar","app.jar"]
---

Docker Image

A Docker image is the blueprint for your container.

It contains the app code and dependencies required to run the project.

Containers are created from images.

---

Docker Architecture

Docker Client

When you type commands like: 

docker ps
docker run

you are using the Docker client. It simply sends instructions to the Docker daemon.

---

Docker Daemon

The Docker daemon runs in the background and performs the actual work.

  • Build images
  • Run containers
  • Manage containers
---

Docker Registry

A Docker registry stores Docker images.

The default public registry is Docker Hub.

It works similar to GitHub but for container images.

Example: 


docker pull moosasadiq18/chess-app

docker run -it moosasadiq18/chess-app
---

Basic Docker Commands

These are the most common commands developers use.

Check Docker version

docker --version

List running containers

docker ps

List all containers

docker ps -a

Pull image from Docker Hub

docker pull nginx

Run a container

docker run -p 8080:80 nginx

Build image from Dockerfile

docker build -t my-app .

Run container from image

docker run -p 8080:8080 my-app

Stop container

docker stop container_id

Remove container

docker rm container_id
---

How to Containerize a Java Project

Basic workflow:


# build project
mvn clean package

# build docker image
docker build -t java-app .

# run container
docker run -p 8080:8080 java-app
