The Only Docker Guide You Need

The actual problem

Before I go into the details of Docker, I want to first explain the problem developers faced that led to its creation.

Suppose you are a backend developer working on a Node.js project and you want your teammate who is a frontend developer to run the service. You send him the repository link.

What usually happens is that he may face issues running that service because the required backend dependencies are not installed on his system.

Even if he installs them, he might still face version conflicts.

Example:


Your service uses an older version of Node.js
But your teammate installs the latest version

This leads to the classic developer problem: 

Works on my machine

Use virtual machines?

Before containerization became popular, Virtual Machines were used to solve this problem.

The typical workflow looked like this:

  • Create a VM with its own operating system
  • Install and configure all dependencies
  • Send the VM file to your teammate

The VM file usually had extensions like:


.vmx
.qcow2

Your teammate would run that VM and the project would work without dependency conflicts.

Wait, no need for containers then...?

Here is where the problem with VMs appears.

A VM carries its own operating system, its own kernel, libraries, and components even if your application never uses them.

A typical VM with a project setup can easily be: 

20GB - 50GB

Running it would also consume a large amount of RAM.

Containers solve the same problem but take a much lighter approach.

  • Containers share the host operating system kernel
  • No separate OS is required
  • Images are usually MBs instead of GBs

So instead of sending a 20GB VM, you might only send a 200MB container image.

That is why containers became the preferred solution.

So… What Even Is Docker?

Think of Docker as a box that contains everything your application needs to run.

Every time you run it, Docker creates a container instance from that box.

Dockerfile

A Dockerfile contains instructions that define how an image should be built.

It specifies things like:

  • Base image
  • Dependencies
  • Commands to run

Docker Image

A Docker image is the blueprint for a container.

It contains the application and all the dependencies required to run it.

Containers are created from images.

Architecture of Docker

1. Docker Client

When you type commands like: 


docker ps
docker run

you are interacting with the Docker client. It simply sends your commands to the Docker daemon.

2. Docker Daemon

The Docker daemon is the engine running in the background.

It performs tasks such as:

  • Building images
  • Running containers
  • Managing containers

You usually do not interact with it directly.

3. Docker Registry

A Docker registry stores Docker images.

Docker Hub is the default public registry.

It works similar to GitHub but for container images.

Example:


docker pull moosasadiq18/chess-app
docker run -it moosasadiq18/chess-app

Basic Docker Commands

These are some commonly used commands developers should know.

Check Docker version

docker --version

List running containers

docker ps

List all containers

docker ps -a

List images

docker images

Build image from Dockerfile

docker build -t my-image .

Run a container

docker run my-image

Stop a container

docker stop container_id

Remove a container

docker rm container_id

How to containerize a simple Java project

Typical workflow:


# build the project
mvn clean package

# build docker image
docker build -t java-app .

# run container
docker run java-app
